# Sprint Evaluation Rubric

Every sprint proposal is scored against these five dimensions. Scores are 1–5 per dimension; a proposal with total ≥ 18 and no dimension below 3 is eligible for acceptance, subject to maintainer capacity.

This rubric is a public commitment, not a negotiation. If you disagree with a score, open a discussion — the rubric evolves with the lab.

## 1. Relevance (weight: ×1)

Does this advance Zcash application space beyond private payments?

| Score | Signal                                                               |
| ----- | -------------------------------------------------------------------- |
| 5     | Directly targets one of the seven verticals with a clear, unmet need |
| 4     | Strong fit to a vertical, partial overlap with existing work         |
| 3     | Adjacent to a vertical or cross-cutting utility                      |
| 2     | Tangential — touches Zcash but not application space                 |
| 1     | Off-topic for this lab                                               |

## 2. Feasibility (weight: ×1)

Can a small team ship a working prototype in 2–4 weeks?

| Score | Signal                                                        |
| ----- | ------------------------------------------------------------- |
| 5     | Clear deliverable, known primitives, scoped to fit the window |
| 4     | Ambitious but achievable; known risks called out              |
| 3     | Plausible, some unknowns                                      |
| 2     | Scope creep risk or missing primitives                        |
| 1     | Research problem, not a sprint                                |

## 3. Team (weight: ×1)

Can the proposers actually execute?

| Score | Signal                                                                     |
| ----- | -------------------------------------------------------------------------- |
| 5     | Track record in Zcash or closely related ZK/privacy work                   |
| 4     | Relevant experience in one of the domains (crypto, credit, identity, etc.) |
| 3     | Strong generalist engineers, willing to learn the domain                   |
| 2     | Light on relevant experience, needs mentoring                              |
| 1     | No demonstrated ability to ship                                            |

## 4. Defensibility (weight: ×1)

Is this hard to clone, meaningful if it ships, and not already solved?

| Score | Signal                                               |
| ----- | ---------------------------------------------------- |
| 5     | Novel combination of primitives; opens a new surface |
| 4     | Meaningful extension of existing work                |
| 3     | Useful consolidation or reference implementation     |
| 2     | Marginal differentiation from existing tools         |
| 1     | Duplicates an existing, maintained project           |

## 5. Handoff path (weight: ×1)

After the sprint, where does this go?

| Score | Signal                                                                  |
| ----- | ----------------------------------------------------------------------- |
| 5     | Clear path to ZCG, ZF Grants, or independent funding with named contact |
| 4     | Strong handoff candidate; path requires some shaping                    |
| 3     | Useful open-source artifact even without follow-on funding              |
| 2     | Handoff unclear; risk of orphaned code                                  |
| 1     | No plan for what happens after the sprint                               |

## Scoring example

A proposal scoring 5 + 4 + 4 + 5 + 4 = **22** with no dimension below 3 is strong. A proposal scoring 5 + 2 + 4 + 5 + 4 = **20** but with a 2 in feasibility is rejected or returned for rescoping.

## What we do not score

- Market size
- Token economics
- Community size
- Marketing plan

We ship code. Markets come later.
